# 写入目标：把角色系统放到哪里

本文说明如何把 `virtual-character-realism-design-zh` 整理出的内容写入不同系统。

它不绑定任何软件、字段名或角色卡规范。无论目标是角色卡、世界书、记忆、预设、状态规则、用户画像，还是某个平台的自定义模块，都先判断内容性质，再决定写入位置。

## 1. 写入前先做分流

不要把所有内容都塞进角色卡。

先判断每段内容的功能：

```text
它是在回答“这个人有哪些已确定事实”吗？ -> 基础信息 / 事实层
它是在回答“这个人是谁”吗？ -> 角色核心
它是在回答“这个世界如何限制他”吗？ -> 世界约束
它是在回答“过去发生过什么并影响以后”吗？ -> 长期记忆
它是在回答“AI 应该怎么演”吗？ -> 角色扮演规则 / 预设
它是在回答“关系如何随互动变化”吗？ -> 关系变化 / 状态规则
它是在回答“用户输入应该怎样被理解”吗？ -> 用户画像
它还没确定、太抽象或会误导模型吗？ -> 暂不写入
```

写入目标不是格式完整，而是让 AI 更稳定地把角色写稳。

### 1.1 基础信息 / 事实层

基础信息适合放在角色卡开头，或任何系统里最先被读到的位置。

适合写入：

- 姓名、年龄、身份、当前处境。
- 与用户或关键对象的关系定位。
- 能被看见的外貌特征。
- 已经发生、并确实塑造了角色的背景事实。
- 重要关系的具体互动位置。

不适合写入：

- 性格标签。
- “这让他变得……”这类作者总结。
- 文学比喻。
- 行为习惯和身体动作的长期模式。
- 尚未确认的推测。

写法建议：

```text
基础信息只写事实，不写性格。外貌写看得见的东西，背景写具体发生了什么，关系写互动的画面。性格的展开放在角色底色、本性与选择、关系变化和记忆里。
```

## 2. 角色核心

角色核心回答：

```text
这个人是谁，他为什么会这样选？
```

适合写入：

- 角色一句话轮廓。
- 当前身份和处境。
- 角色底色。
- 本性与选择。
- 说话方式。
- 代表性动作。
- 与用户或关键对象的关系位置。
- 三重面相的简短版本。
- 晕染里最有代表性的动作。
- 防写偏说明中的关键禁区。

不适合写入：

- 长篇世界背景。
- 已经发生的互动流水账。
- 大量抽象形容词。
- 平台演出规则。
- 尚未确认的推测。

写法建议：

```text
角色核心要短而硬。每一条都应该能改变角色下一次反应。
```

示例结构：

```text
角色核心：
- 他最稳定的底色是……
- 他在冲突中通常先保护……
- 他表达需要时常常……
- 他最容易被写偏成……，但正确方向是……
```

## 3. 世界约束

世界约束回答：

```text
这个人活在哪里，哪些外部事实限制了他？
```

适合写入：

- 社会规则。
- 家族、组织、职业、阶层、制度。
- 地点与空间规则。
- 禁忌、代价、资源限制。
- 旧关系和外部压力。
- 角色不能随便做某事的原因。
- 世界事实如何触发角色的面相、底色或防御。

不适合写入：

- 角色内心独白。
- 角色已经发生的私人记忆。
- 纯百科条目。
- 与角色选择无关的宏大设定。

写法建议：

```text
世界约束不要写成百科。写事实、限制、代价和触发。
```

示例结构：

```text
世界约束：
- 在此环境中，角色不能随意……
- 如果他违反这条规则，代价是……
- 这个地点会让他启动……
```

## 4. 长期记忆

长期记忆回答：

```text
已经发生过什么，并且会改变角色以后怎么反应？
```

适合写入：

- 已发生事件。
- 用户与角色之间的关键互动。
- 承诺、误解、拒绝、和解。
- 羞耻、亏欠、偏好、旧话题。
- 关系阶段变化的证据。
- 会改变后续语气、距离、称呼或选择的细节。

不适合写入：

- 没发生过的未来计划。
- 大段背景说明。
- 不会影响后续反应的流水账。
- 每轮都要重复的固定人格规则。

写法建议：

```text
记忆不是日记。只写会改变之后反应的事实。
```

示例结构：

```text
长期记忆：
- 已发生：……
- 对角色的影响：……
- 后续触发：……
- 不要每轮复读：……
```

## 5. 角色扮演规则 / 预设

角色扮演规则回答：

```text
AI 应该怎么演，怎么不演错？
```

适合写入：

- 不替用户说话。
- 不替用户决定动作或感受。
- 不把日常小事写成高潮戏。
- 不反复复读痛点。
- 不用旁白替角色讲心理论文。
- 亲密、身体性、欲望必须服务人格、边界和关系变化。
- 多角色时保持声音分开。
- 回复节奏、叙事视角、动作密度。
- 防写偏说明。
- 去 AI 腔规则。

不适合写入：

- 角色本人的稳定人格事实。
- 世界事实。
- 已发生的事件记忆。
- 需要动态变化的关系阶段。

写法建议：

```text
角色扮演规则要短、明确、可执行。不要写成口号。
```

示例结构：

```text
角色扮演规则：
- 不替用户行动或说话。
- 日常互动只允许轻微流露，不自动升级成崩溃。
- 角色表达需要时优先绕路，不直接剖白。
- 亲密场景必须保留停顿、边界和关系后果。
```

## 6. 关系变化 / 状态规则

关系变化回答：

```text
互动后，角色是否因此不同？
```

适合写入：

- 信任阶段。
- 戒备变化。
- 亲密推进。
- 依恋什么时候露出来。
- 羞耻加重或减轻。
- 冲突、拒绝、和解后的后效。
- 阶段触发条件。
- 每个阶段的语言、距离、动作、误读变化。
- 不变量。

不适合写入：

- 固定人格说明。
- 世界背景。
- 单次事件流水账。
- 没有可观察差异的状态名。

写法建议：

```text
状态不是装饰。每个阶段都必须改变角色之后的反应。
```

示例结构：

```text
关系变化：
- 低信任：……
- 中信任：……
- 高信任：……
- 下降触发：……
- 不变量：……
```

## 7. 用户画像

用户画像回答：

```text
AI 应该怎样理解用户的输入方式和关系位置？
```

先判断用户画像是哪一种。

```text
AI 需要续写或扮演用户 -> 简化用户角色卡。
AI 不应该扮演用户，只需要回应用户 -> 行为翻译手册。
```

适合写入：

- 用户在故事里的身份或关系位置。
- 用户常用短句、沉默、动作、命令或回避方式。
- 哪些用户行为不代表拒绝。
- 哪些用户行为不代表同意。
- 用户边界。
- 角色容易怎样误读用户。
- 角色应该怎样确认，而不是替用户决定。

不适合写入：

- 替用户安排完整性格。
- 替用户决定行动。
- 替用户表达感受。
- 抢走角色主场。

写法建议：

```text
用户画像更像行为翻译手册，不是用户角色卡。
```

如果是“不抢话”的玩法，不要给用户写完整角色底色。AI 不需要演用户，只需要知道“用户这样做通常是什么意思”。例如：

```text
用户沉默时多半是在想措辞，不自动理解成冷暴力。
用户说“过来”通常是熟人式关心，不自动理解成支配。
用户短句不等于敷衍，也不等于同意。
```

如果是“需要 AI 续写用户”的玩法，可以写简化用户角色卡，但仍要短，并加禁止反复提及的说明。

示例结构：

```text
用户画像：
- 用户的短句通常不代表……
- 用户沉默时，角色可以轻微误读，但不能自动……
- 用户明确拒绝时，角色必须……
```

## 8. 物象、意境、心象怎么写

### 8.1 物象

物象适合写进：

- 角色核心。
- 记忆。
- 世界约束。
- 场景触发规则。

写法：

```text
物象必须能被看见、听见、碰到。它不是装饰。
```

### 8.2 意境

意境适合写进：

- 角色核心的简短锚点。
- 角色扮演规则。
- 场景风格说明。

写法：

```text
意境帮助统一氛围，但不能替代事实和动作。
```

### 8.3 心象

心象适合写进：

- 角色核心的内在锚点。
- 关键记忆。
- 少量角色扮演规则。

写法：

```text
心象要少用，避免让角色满口诗意或过度自我解释。
```

## 9. 亲密、身体性与欲望怎么写

亲密、身体性、欲望、羞耻、依恋不是特殊模式，而是人格的一部分。

分流方式：

- 角色为什么渴望或抗拒亲密 -> 角色核心。
- 亲密里的确认需求、羞耻、防御 -> 本性与选择。
- 明确拒绝后必须停下 -> 角色扮演规则。
- 亲密后关系阶段改变 -> 关系变化。
- 已发生的亲密事件和后效 -> 长期记忆。
- 世界或身份限制亲密 -> 世界约束。

写法建议：

```text
不要把欲望写成动作清单。写确认、边界、羞耻、身体反应和关系后果。
```

**具体写法**（驱动类型、衍生、身体反应、事后状态、防写偏）见 [intimacy.md](references/intimacy.md)。

## 10. 不建议写入的内容

以下内容先不要写入正式系统：

- 作者未确认的推测。
- 只有抽象词，没有行为后果的描述。
- 与角色选择无关的世界百科。
- 太像 AI 总结的漂亮句子。
- 会让角色变成单一痛点的重复内容。
- 只适合作者自己参考、不适合喂给模型的分析过程。
- 可能侵犯他人版权、来源不明或可识别的外部素材。

可以暂存为：

```text
待确认：
仅供作者参考：
暂不写入：
```

## 11. 平台容量不同时怎么压缩

### 11.1 低容量平台

优先保留：

1. 角色底色。
2. 本性与选择。
3. 关系位置。
4. 防写偏说明。
5. 角色扮演规则。

删减：

- 长背景。
- 低频物象。
- 复杂关系阶段。
- 不影响选择的世界百科。

### 11.2 中容量平台

可以加入：

- 三重面相。
- 晕染画面。
- 用户画像。
- 简短世界约束。
- 简短关系变化。

### 11.3 高容量平台

可以拆分：

- 角色核心放角色卡。
- 世界约束放世界书。
- 已发生事件放记忆。
- 扮演纪律放预设。
- 阶段推进放状态规则。
- 用户输入理解放用户画像。

高容量不等于全塞。仍然要删掉不会改变角色反应的内容。

## 12. 写入后怎么测试

写入后用短场景测试。

至少测三类：

```text
日常场景：角色是否仍有底色，而不是只会演痛点？
压力场景：角色是否按本性与选择反应？
关系场景：角色是否记得用户画像和关系变化？
```

可选测试：

- 用户沉默。
- 用户拒绝。
- 用户靠近。
- 用户要离开。
- 角色被看穿。
- 亲密后普通对话。
- 世界规则触发。

测试后输出：

```text
# 写入测试反馈

## 生效的内容

## 没有生效的内容

## 写偏风险

## 需要移动写入位置的内容

## 需要删减或压低的内容
```

## 13. 通用输出模板

当用户要求“帮我分配写入位置”时，输出：

```text
# 写入分流建议

## 角色核心

## 世界约束

## 长期记忆

## 角色扮演规则 / 预设

## 关系变化 / 状态规则

## 用户画像

## 物象 / 意境 / 心象

## 亲密、身体性与欲望

## 暂不建议写入

## 需要作者确认
```

当用户已经指定平台或字段时，先按以上逻辑分流，再映射到目标字段。不要为了适配字段而改变事实边界。
